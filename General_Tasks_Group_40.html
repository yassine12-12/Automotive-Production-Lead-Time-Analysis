a) How is the logistics delay distributed? Justify your choice with statistical tests and briefly describe your approach. (2 points)¶

Approach.
The logistics delay is centered around 6 days (mean = 6.08, median = 6, mode = 6) with low variability (std = 1.01, IQR = 5–7). Most values fall between 5 and 7 days, while a few extreme cases extend up to 14 days. The histogram shows a strong peak at 6. The boxplot further reveals slight right skewness: the median lies slightly closer to the lower quartile, the upper whisker is longer than the lower whisker, and there are several outliers above 10 days, all of which are typical signs of a right-skewed distribution. This observation is confirmed numerically by the positive skewness (0.57). The kurtosis (0.63) suggests tails that are somewhat heavier than those of a Normal distribution.

Normality was tested using the Shapiro–Wilk and Kolmogorov–Smirnov tests. In both cases, the null hypothesis of normality was rejected (p < 0.05).

Conclusion.
The logistics delay is not normally distributed. Although it is tightly concentrated around 6 days with low dispersion, the data shows slight right skewness and heavier tails, causing it to deviate from a Normal distribution.

On average, goods take 6.08 calendar days from shipping to arrival.
If weekends are excluded, the mean delay drops to 4.34 business days.
This means that 1.74 days (≈28.6%) of the observed delay is due solely to weekends rather than actual transportation or processing.

An alternative to reporting delays in calendar days is to use business days, which provides a more realistic measure of operational performance. In practice, it can also be useful to report both values: calendar days to capture the customer perspective, and business days to evaluate internal logistics efficiency. The approach may be further refined by excluding public holidays in addition to weekends, yielding an even more accurate estimate of effective processing times.

c) Visualize the distribution appropriately by displaying the histogram and density function using “plotly.” Describe how you selected the size of the bins. (2 points)¶

At first, we determined the number of bins using the Freedman–Diaconis rule, since it adapts the bin width to both the spread of the data (via the interquartile range) and the sample size. However, this resulted in a very large number of bins (≈186) with a very narrow width, which made the histogram appear overly spiky and difficult to interpret. Because our data represent logistics delays measured in whole days and range only between 3 and 14, we decided to replace the Freedman–Diaconis choice with 1-day bins aligned to the integer values. This approach better matches the discrete nature of the data, produces a clearer visualization, and allows for a more intuitive interpretation of the distribution.

References: [https://www.statology.org/choosing-the-optimal-bin-size-for-your-histogram/]

# Freedman–Diaconis bin width 
n = delay.size
data_min, data_max = float(delay.min()), float(delay.max())
q25, q75 = np.percentile(delay, [25, 75])
iqr = q75 - q25

h = 2 * iqr * (n ** (-1/3))   # FD bin width
bins = max(int(np.ceil((data_max - data_min) / h)), 1)

Task 2: Data Storage in Separate Files (2 Points)¶

Question: Why does it make sense to store the available data in separate files instead of saving everything in one large table?
Name at least four benefits. What is this database structure called?

Storing data in separate files or tables rather than in one large table follows the principles of relational database design and data normalization. Each table represents a specific entity (e.g., vehicles, components, logistics, registrations) while relationships between entities are maintained through primary keys and foreign keys, as illustrated by the Entity–Relationship (ER) model in the lecture 6. This approach reduces redundancy, ensures integrity and consistency, improves performance, and increases scalability and modularity.

Key Benefits¶

-Reduced Redundancy: Each fact is stored once (e.g., components linked by ID), saving space.

-Integrity & Consistency: Updates in one table propagate via keys, avoiding contradictions.

-Performance: Smaller tables allow faster queries, indexing, and optimization.

-Maintainability: changes affect only relevant tables, not the entire dataset

Additionally it enables better security (table-level access) and flexibility (different file formats:CSV, TXT, Excel).

Database structure: This is a Relational Database Schema

import os
import pandas as pd

# Configuration
ZULASSUNGEN_FILE = os.path.join(DATA_PATH, "Zulassungen", "Zulassungen_alle_Fahrzeuge.csv")
FAHRZEUG_DIR     = os.path.join(DATA_PATH, "Fahrzeug")
KOMPONENTE_DIR   = os.path.join(DATA_PATH, "Komponente")

# ----------------  Adelshofen vehicles ----------------
zul = pd.read_csv(ZULASSUNGEN_FILE, sep=";", encoding="utf-8")
adel_ids = set(
    zul.loc[zul["Gemeinden"].str.contains("Adelshofen", case=False, na=False), "IDNummer"]
      .astype(str)
      .unique()
)
print(f"Vehicles in Adelshofen: {len(adel_ids):,}")

# ----------------  Vehicle → component IDs ----------------
component_types = {}  

for fname in sorted(f for f in os.listdir(FAHRZEUG_DIR) if f.startswith("Bestandteile_") and f.endswith(".csv")):
    df = pd.read_csv(os.path.join(FAHRZEUG_DIR, fname), sep=";", encoding="utf-8")
    if "ID_Fahrzeug" not in df.columns:
        continue
    df["ID_Fahrzeug"] = df["ID_Fahrzeug"].astype(str)
    m = df["ID_Fahrzeug"].isin(adel_ids)
    if not m.any():
        continue

    # Collect all component ID columns for matched vehicles
    for col in df.columns:
        if col.startswith("ID_") and col != "ID_Fahrzeug":
            ids = df.loc[m, col].dropna().astype(str).unique()
            if ids.size:
                component_types.setdefault(col, set()).update(ids)
print(f"Unique component IDs (sum over types): {sum(len(s) for s in component_types.values()):,}")

# ---------------- Components → subparts (count ID_T16) ----------------
total_t16 = 0
# Collect all component IDs into a single set for simpler matching
all_component_ids = set()
for component_set in component_types.values():
    all_component_ids.update(component_set)

for fname in sorted(f for f in os.listdir(KOMPONENTE_DIR) if f.startswith("Bestandteile_Komponente_") and f.endswith(".csv")):
    df = pd.read_csv(os.path.join(KOMPONENTE_DIR, fname), sep=";", encoding="utf-8")
    
    if "ID_T16" not in df.columns:
        continue  # Skip files without T16 data

    # Find any ID column that matches our Adelshofen components  
    mask = pd.Series(False, index=df.index)
    for col in df.columns:
        if col.startswith("ID_") and col != "ID_T16":  
            col_matches = df[col].astype(str).isin(all_component_ids)
            if col_matches.any():
                mask |= col_matches
                print(f"Found {col_matches.sum()} matches in column '{col}' of {fname}")
    
    # Count T16 parts for matched components
    if mask.any():
        total_t16 += df.loc[mask, "ID_T16"].notna().sum()



print(f"T16 parts identified in Adelshofen vehicle components: {total_t16:,}")

import pandas as pd
from pathlib import Path

# Defining the target PART and DATE
PART = "K5-112-1122-79"
DATE = pd.Timestamp("2010-08-11")

# Step 1: read ALL Fahrzeug CSVs automatically 
fahrzeug_path = DATA_PATH / "Fahrzeug"
parts_files = list(fahrzeug_path.glob("*.csv"))   # all CSV files in Fahrzeug/
parts = pd.concat([pd.read_csv(f, sep=";") for f in parts_files], ignore_index=True)

# Step 2: find vehicle(s) with the target body part
veh_ids = parts.loc[parts["ID_Karosserie"] == PART, "ID_Fahrzeug"].astype(str).unique()

# Step 3: check registrations 
regs = pd.read_csv(DATA_PATH/"Zulassungen"/"Zulassungen_alle_Fahrzeuge.csv",

regs["Zulassung"] = pd.to_datetime(regs["Zulassung"], errors="coerce")

valid = regs[regs["IDNummer"].astype(str).isin(veh_ids) & (regs["Zulassung"] <= DATE)]
latest = valid.sort_values(["IDNummer","Zulassung"]).drop_duplicates("IDNummer", keep="last")
